## OOM（显存溢出） 解决方案 

在大型语言模型（LLM）的训练和推理过程中，显存溢出（Out of Memory, OOM）是最常见且最棘手的问题之一。由于 LLM 动辄拥有数十亿甚至上千亿的参数，加上庞大的激活值（Activations）、优化器状态（Optimizer States）和 KV Cache，显存管理变得至关重要。

将从下方几个维度阐述：
- 算法层面
- 工程层面
- SGLang和vLLM

### 一、算法层面：Gradient Checkpointing（梯度检查点）
在模型训练中，显存占用大头不仅仅是模型参数，更多的是**前向传播（Forward Pass）时产生的激活值**。为了在反向传播（Backward Pass）时计算梯度，标准流程会将所有层的激活值都保存在显存中，这在层数极深的 LLM 中极易导致 OOM。

**Gradient Checkpointing 的核心思想是“以时间换空间”**

1. **选择性保存**：在前向传播时，不再保存所有中间层的激活值，而是只保存其中一部分特定的层（即 Checkpoints / 检查点）。

2. **按需重计算**：在反向传播时，当需要用到未保存的激活值来计算梯度时，系统会从距离它最近的那个“检查点”开始，重新执行一次前向传播来临时计算出这些缺失的激活值。

3. **收益与代价**：这种算法可以将激活值占用的显存从 $O(N)$ ( $N$ 为层数)大幅降低到 $O(\sqrt N)$ ，从而极大地缓解了 OOM 问题。代价是增加了额外的计算量（因为部分前向传播被执行了两次），通常会导致整体训练时间增加 20%~30% 左右。

### 二、工程层面：Offloading 与 Mixed Precision
工程层面的优化通常通过压榨硬件资源的潜力或改变数据表示方式来释放显存。

1. **Offloading（卸载到 CPU、NVMe）**
    
    GPU 显存极其昂贵且容量有限（例如 H100 通常为 80GB），而系统内存（CPU RAM）和固态硬盘（NVMe SSD）的容量要大得多且便宜。
    - **核心机制**： Offloading 技术（如 DeepSpeed 的 ZeRO-Offload 及其后续版本）将暂时不参与计算的数据（通常是优化器状态、梯度，甚至是模型参数）从 GPU 显存“卸载”到 CPU 内存或 SSD 中。
    - **按需加载 (Just-in-Time)**： 当 GPU 需要进行计算时，再通过 PCIe 总线将需要的数据实时拷贝回 GPU。计算完成后，再将产生的新状态写回 CPU/SSD。
    - **效果**： 这种方法让原本只能在多张 GPU 上训练的模型，可以在单张或少量 GPU 上运行，彻底打破了 GPU 显存容量的物理限制。瓶颈在于 PCIe 的带宽传输速度，但在 OOM 面前，这是一种极其有效的妥协。

2. **Mixed Precision（混合精度）**

    传统的深度学习通常使用 FP32 （32位单精度浮点数），这会消耗大量显存。
    - **核心机制**： 混合精度训练（如使用 FP16 或 BF16）在模型的前向和反向传播过程中使用 16 位浮点数。这直接将参数、激活值和梯度的显存占用**减半**。
    - **主权重保留 (Master Weights)**： 为了防止 16 位计算带来的精度溢出（Overflow）或下溢（Underflow）导致模型无法收敛，优化器内部会保留一份 FP32 的“主权重”进行参数更新。
    - **效果**： 不仅大幅降低了 OOM 的风险，还能充分利用现代 GPU 的 Tensor Core 进行加速，使得训练和推理速度成倍提升。目前 BF16（Brain Float 16）因其更大的动态范围，已成为 LLM 训练的标配。

### 三、SGLang 的 DP (Dynamic Programming) 算法优化

注：SGLang 是一个专门为 LLM 推理设计的高效执行引擎。在推理阶段，OOM 的主要元凶不再是梯度，而是随着并发请求和上下文长度急剧膨胀的 **KV Cache**。SGLang 的核心创新是 RadixAttention（基数树注意力），而动态规划（DP）算法在其中扮演了智能调度的关键角色。

**SGLang 中基于 DP 算法优化的解决方案主要体现在以下几个方面：**

1. **最优 KV Cache 前缀匹配与复用 (Prefix Sharing)**：
当处理复杂的 Prompt 模板（如 Few-shot、Tree-of-Thought 或多轮对话）时，会有大量重复的前缀。SGLang 使用 Radix Tree 来全局管理 KV Cache。为了在极其有限的显存中最大化吞吐量，系统可以通过 DP 算法来计算**最优的请求调度策略**。DP 算法可以评估不同请求的前缀重合度，将具有高重复度的请求安排在同一批次或相邻时间点执行，从而最大程度共享显存中的 KV Cache。
- **智能驱逐策略 (Eviction Policy) 避免 OOM**：
当 GPU 显存即将耗尽（面临 OOM 危机）时，必须将一部分 KV Cache 从显存中驱逐。传统的 LRU（最近最少使用）策略过于简单。SGLang 可以利用 DP 算法，结合 Radix Tree 的拓扑结构、节点的访问频率以及未来的预测使用概率，计算出一条**代价最小的驱逐路径**。这样不仅避免了当前的 OOM，还能确保被驱逐的节点在未来重新计算时的代价（Recomputation Cost）降到最低。
- **长文本分块处理的最优解 (Optimal Chunking)**：
在处理超长上下文（如几十万 token）时，单次处理极易直接导致 OOM。SGLang 及其底层的调度机制可以利用动态规划算法，将长文本切分为最优的 Chunk（块）序列。DP 算法会根据当前可用的剩余显存（Memory Budget）和硬件算力限制，计算出每一刀应该切在哪里，使得既不触发 OOM，又能维持最高的并行计算效率。
