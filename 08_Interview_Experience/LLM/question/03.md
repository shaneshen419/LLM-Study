## 3. 你用了 swift 参数怎么设置的，有 2 个 node 你如何分配你的训练参数？ 

### （a）核心概念：双机训练架构
假设你有两个节点（机器）：

- Node 0 (Master): IP 192.168.1.100, 有 8 张卡。
- Node 1 (Worker): IP 192.168.1.101, 有 8 张卡。
- 总卡数 (World Size): 16 张。

你的目标是让这 16 张卡像一个整体一样工作。

### （b）启动方式与环境配置 (最关键的一步)
Swift 底层依赖 PyTorch Distributed (torchrun 或 accelerate)。双机训练不能直接简单的运行 swift sft，必须配置环境变量并使用分布式启动器。

你需要分别在两台机器上运行命令。

1. **Node 0 (主节点) 的启动命令**
```python
# 环境变量设置
export MASTER_ADDR=192.168.1.100  # 主节点 IP
export MASTER_PORT=29500          # 通信端口
export NPROC_PER_NODE=8           # 每台机器的卡数
export NNODES=2                   # 总机器数
export NODE_RANK=0                # 当前机器编号 (主节点为0)

# 启动命令 (使用 torchrun 封装 swift)
torchrun \
    --nproc_per_node=$NPROC_PER_NODE \
    --nnodes=$NNODES \
    --node_rank=$NODE_RANK \
    --master_addr=$MASTER_ADDR \
    --master_port=$MASTER_PORT \
    swift sft \
    --model_type qwen1half-72b-chat \
    --sft_type lora \
    --deepspeed default-zero3 \
    ... (其他参数)

```

2. **Node 1 (从节点) 的启动命令**

**注意**： 除了 NODE_RANK 变为 1，其余环境变量必须与 Node 0 完全一致（尤其是 MASTER_ADDR 必须指向 Node 0）。
```python
export MASTER_ADDR=192.168.1.100  # 依然指向 Node 0 的 IP
export MASTER_PORT=29500
export NPROC_PER_NODE=8
export NNODES=2
export NODE_RANK=1                # !!! 这里改为 1 !!!

torchrun \
    --nproc_per_node=$NPROC_PER_NODE \
    --nnodes=$NNODES \
    --node_rank=$NODE_RANK \
    --master_addr=$MASTER_ADDR \
    --master_port=$MASTER_PORT \
    swift sft \
    ... (参数必须与 Node 0 完全一致)

```

### （c）训练参数分配策略 (Strategy Allocation)
你有 2 个节点，如何分配参数取决于你的 **模型大小 和 显存大小**。这里主要分两种情况：

1. **场景 A：模型很大，单卡放不下 (例如 72B 模型, 24G/40G/80G 显存)**

    此时必须使用 DeepSpeed ZeRO-3 或 FSDP 将参数切分到所有 16 张卡上。
    - **Swift 参数设置**：
        - --deepspeed default-zero3: 这是一个“一键开关”。它会自动启用 ZeRO Stage 3，把模型参数、梯度、优化器状态切分到 2 台机器的 16 张卡上。
    - **原理**：
        - 每张卡只存 1/16 的参数。
        - 训练时，Node 0 和 Node 1 之间会通过网络疯狂交换数据（All-Gather）。
    - **注意**: 此时两台机器之间的网络带宽（如 InfiniBand 或 万兆以太网）会成为瓶颈。如果网速慢，训练会非常卡。

2. **场景 B：模型较小，单卡放得下 (例如 7B/14B 模型, 80G A100)**

    此时可以使用 **DDP (ZeRO-0/1/2)**。
    - **Swift 参数设置**：
        - --deepspeed default-zero2: 只切分优化器状态和梯度，参数每张卡存一份完整的。
        - 或者不加 deepspeed 参数（默认 DDP），但这在微调大模型时很少见。
    - **优势**: 通信量比 ZeRO-3 小很多，训练速度更快。

### （d） 关键超参数 (Hyperparameters) 的调整
当从单机扩展到双机（GPU 数量翻倍）时，你需要重新计算以下参数：

1. **Batch Size**与**梯度累加（Gradient Accumulation）**

    在分布式训练中，存在一个核心的公式：

    Global Batch Size = per_device_train_batch_size × world_size(总卡数) × gradient_accumulation_steps

    当把卡数从8张增加到 16 张时，如果不修改其他参数，你的Global Batch Size 会直接翻倍。这可能导致显存溢出（OOM）或影响收敛。

    - **调整策略**：如果你希望保持和单机8卡时相同的Global Batch Size（以复现单机训练的收敛曲线），需要将 gradient_accumulation_steps **减半**。

    - **Swift参数示例**：

    ```python
    # 假设单机 8 卡时是：--batch_size 2 --gradient_accumulation_steps 4 (全局 BS = 2*8*4 = 64)
    # 现在双机 16 卡，为了保持全局 BS 还是 64，你需要改成：
    --batch_size 2
    --gradient_accumulation_steps 2  # 4 减半为 2
    ```

2. **学习率**

    根据深度学习中的“线性缩放法则”：如果你的Global Batch Size增加了，你的学习率通常会需要相应增加（通常是等比例扩大，或者乘以扩大倍数的平方根）。

    - **调整策略**：
        - 如果你通过调整梯度累加保持了全局 Batch Size 不变，那么学习率 **不需要** 修改。
        - 如果你让全局 Batch Size 翻倍了（比如想利用 16 张卡加速吃更多数据），那么你可能需要适度调高 <span style="background-color: gray;">--learning_rate</span> （例如从 1e-5 调整到 1.5e-5 或 2e-5），并配合适当的<span style="background-color: gray;">--warmup_ratio</span> 避免训练初期不稳定。

3. **评估、保存和日志步数(Steps)**

    由于总卡数翻倍，如果在相同数据量下，走完一个Epoch所需的物理步数（Steps）会大幅减少。

    - **调整策略**：你需要相应缩小 --sace_steps，--eval_steps，和 --logging_steps 的值，否则可能会出现“好不容易训完一个Epoch却还没来得及保存/打印日志”的情况。
    - **Swift参数示例**

    ```python
    --logging_steps 10      # 之前可能是 20
    --eval_steps 100        # 之前可能是 200
    --save_steps 100        # 之前可能是 200
    ```

4. **分布式通信超时设置（NCCL Timeout）**

    双机训练引入了节点的网络通信，如果两台机器之间的网络存在波动，或者某一层的 ALL-Gather 操作卡顿，很容易除法NCCL 默认的超时报错。

    - **调整策略**：在两台机器的启动脚本中，额外增加 NCCL 的超时环境变量，防止因为短暂的网络拥塞导致整个训练任务崩溃。
    - **环境配置示例**：
    ```python 
    export NCCL_TIMEOUT=7200         # 延长到 2 小时 (7200秒)
    export NCCL_IB_DISABLE=0         # 如果有 InfiniBand 网络，确保启用
    export NCCL_DEBUG=INFO           # 如果遇到多机卡死问题，开启 INFO 方便排查网络连通性
    ```
