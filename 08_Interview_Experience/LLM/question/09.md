## 长上下文压缩有哪些方法？

长上下文压缩（Long Context Compression）是解决 Agentic 任务中“显存爆炸（OOM）”和“推理延迟过高”的核心技术。随着模型支持的上下文长度从 32K 飙升到 1M 甚至更长，如何把冗长的历史信息“浓缩”下来，成为了目前的工程重点。

### 一、提示词/文本级压缩(Data/Prompt Level)
在输入模型之前，直接砍掉或浓缩无用的文本（Token）。这种方法最简单，不需要改模型底层代码。

1. **信息熵压缩（如LLMLingua系列）**：
    - **原理**：自然语言中存在大量冗余（比如“的”、“地”、“得”和过渡句）。利用一个小巧的语言模型（如极小的尺寸的基座模型）计算Prompt中每个Token的困惑度。
    - **操作**：保留困惑度高（信息量大、不可预测）的Token，剔除困惑度低（废话）的Token。
    - **效果**：可以再几乎不损失关键语义的情况下，将几十 KB 的文档压缩掉 50% 甚至 70% 的长度。

2. **层次化总结**：
    - **原理**：传统的“总结”会丢失细节。对于 Agent 的长历史，可以采用树状结构的总结。将旧的对话日志按 Chunk 总结，只有最近的 N 轮对话保留原始长文本。

3. **动态 RAG 替代硬塞**：
    - **原理**：不要把几十万字全塞进Prompt。把长文档向量化存入本地的 Vector DB，Agent 需要时通过 Tool Call 进行语义检索（Top-k），把“长上下文问题”转化为“动态短上下文问题”。

### 二、KV Cache压缩
大模型在推理时，每生成一个Token，前面的所有历史特征都要存在显存里（这就是 KV Cache）。这里的压缩主要通过“丢弃”或“量化”来释放显存。

1. **注意力截断与沉淀**：
    - **发现**：模型在计算注意力时，会把极其庞大的注意力分数分配给前几个 Token（如 System Prompt 或开头的特殊符号），这被称为 Attention Sinks（注意力黑洞）。
    - **操作**：永远保留最开头的 4 个 Token 的 KV Cache ，加上一个最近的滑动窗口（比如最近的 1000 个 Token）。中间的几万个 Token 的 KV Cace 阅读后，直接丢弃。
    - **效果**：显存占用从 $O(n)$ 变成常数级，支持“无限长度”的流式对话而不崩。

2. **动态 Token 驱逐**：
    - **原理**：并不是所有词对后续生成都有用。H2O（Heavy-Hitter Oracle）等算法会在生成过程中，实时监控哪些历史 Token 经常被查询（注意力分数高）。
    - **操作**：保留高分 Token（Heavy-Hitters），把低分 Token 的 KV Cache从显存中踢出去。

3. **KV Cache量化**：
    - **操作**：将原本占用巨大显存的 FP16/BF16 KV Cache，实时量化为 INT8、FP8 甚至 INT4、W4A16 格式。
    - **效果**：算力开销微增，但显存占用直接缩小 2 倍到 4 倍，极大缓解 OOM。

### 三、模型架构级压缩
如果你自己预训练或深度微调（SFT）模型，可以通过改变架构来压缩上下文。

1. **稀疏注意力**：
    - 打破 $O(n^2)$ 的全局注意力机制。强制每个 Token 只关注它周围的局部 Token 以及几个特定的全局 Token。

2. **压缩 Token**：
    - 通过特殊的微调，教会模型使用几个特定的 <GIST> token
    - 把前文丢给模型，让模型把前文的全部语义“编码”并“浓缩”到这几个 Token 的隐藏状态中。后续对话只需要带着这几个压缩后的 Token 即可，抛弃前面的原始文本。

3. **Token 合并**：
    - 在 Transformer 的中间层，通过计算 Token 之间的余弦相似度，将语义极其相似的 Token 直接融合（相加取平均）成一个 Token。这在多模态（视觉 Patch 压缩）和长文本中很有效。

### 四、框架与系统级优化

1. **前缀缓存**：
    - 在 Agentic 场景中，许多请求共享一模一样的 System Prompt 或 Tool Definitions。
    - 想 vLLM 或 SGLang 等现代推理引擎，直接复用已经算好的 KV Cache，实现“0”消耗“加载长上下文”。

### 五、操作建议步骤

1. 开启框架层面的 Prefix Caching 和 FlashAttention

2. 开启 KV Cache INT8/FP8 量化。

3. 引入 LLMLingua 剔除冗余 Prompt，或者强制 Agent 使用 RAG 而不是读取全量文本。

4. 采用 StreamingLLM（滑动窗口+注意力沉淀），牺牲一部分超长距离的 历史关联能力，换取系统绝对不 OOM。
