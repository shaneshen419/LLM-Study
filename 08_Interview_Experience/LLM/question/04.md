## 做 grpo 遇到熵崩没有？你是怎么理解的？

GRPO (Group Relative Policy Optimization)因为抛弃了庞大的 Critic（价值模型），主要靠对同一 Prompt 采样一组（Group）答案来计算相对优势（Advantage），极大节省了显存。但随之而来的代价就是训练的不稳定性，其中最典型的就是**熵崩**（Entropy Collapse）。

### 一、什么是 GRPO 的“熵崩”

在强化学习微调（RLHF）中，**策略熵（Policy Entropy）**衡量的是模型输出概率分布的多样性和随机性。

**熵崩**是指在 GRPO 训练过程中，模型的策略熵在短时间内迅速下降，逼近于0.
- 现象表现：模型彻底丧失了探索能力（Exploration）。对于给定的prompt，模型不再考虑多种可能的合理回答，而是极其确信地只输出某一种特定的文本（概率分布变得极其尖锐）。
- 最终结果：模型陷入“模式崩溃（Mode Collapse）”，可能会变成复读机，或者反复输出毫无逻辑但能卡 Reward 模型 Bug 的无意义字符（Reward Hacking）。

### 二、什么原因可能引起 GRPO 的熵崩
GRPO 相比 PPO 移除了庞大的 Critic（价值模型），通过对同一 Prompt 采样一组（Group）输出来计算相对优势（Advantage），这种机制更容易导致熵崩，主要原因如下：

1. **局部优势估计的不稳定性（Group Size过小）**：GRPO 通过计算组内 Reward 的均值和方差来进行 Z-score 标准化。如果采样的组大小（Group Size）太小（如仅为4），局部的均值和方差会非常剧烈地波动。这种高方差的优势估计会产生极端的梯度，瞬间把模型的概率分布推向极端。

2. **KL 乘法约束不足（KL Penalty太弱）**：强化学习微调依赖 KL 散度将当前策略拉向初始参考模型（Reference Model）。如果 KL 惩罚系数设置过低，模型为了最大化 Reward 会毫无顾忌地偏离初始状态，迅速收敛到单一输出路径。

3. **Reward 模型的高分诱导（Reward Hacking）**：如果 Reward 模型不完美（或者基于规则的打分出现极值），某个特定的token 序列意外获得了畸形的高分，模型会贪婪地捕获这个“捷径”，把几乎所有的概率质量都几种在这几个 token 上。

4. **学习了过大**：过大的学习率会导致策略更新步子迈得太大，一次更新就把某些token 的概率推到了接近 100%。

### 三、怎么防范/遇到了怎么避免
如果在训练中监控到 Entropy曲线出现断崖式下跌，或者 KL 三都急剧飙升，可以采取以下手段进行防范和修复：

1. **引入/增大熵惩罚项**：在损失函数中显式加入一项对策略熵的奖励。只要模型的输出分布变得过于尖锐（熵变低），就给它惩罚，强制要求模型保持一定程度的输出多样性。

2. **采用动态 KL 散度控制（Adaptive KL Controller）**：放弃固定的 KL 惩罚系数 $\beta$ 。监控实时的 KL 散度值，如果发现偏离参数模型过快（往往是熵崩的前兆），就自动调大 $\beta$ 值，把模型“硬拽”回来。

3. **调整核心超参数（Group Size & Learning Rate）**：
    - **增大Group Size**：在显存允许的情况下，将Group Size调大（如8 或 16），让相对优势的估算更加平稳、有统计学意义。
    - **降低学习率**：减小Actor模型的学习率，让分布的更新更加平缓。

4. **对 Reward 进行截断与标准化 (Reward Clipping)**：对 Reward 函数的输出进行硬截断（例如限制在 $[-5,5]$ 范围内），防止个别离谱的高分 Reward 产生核弹级别的梯度更新。

5. **调整 Rollout 的采样温度 (Temperature)**：在 GRPO 生成候选回答的阶段，确保设置了合理的 Temperature（如 0.7 - 1.0）和 Top-p，保证喂给模型去计算 Reward 的样本本身是具备多样性的。如果采样本身都是完全一致的，优势计算就会失效。
