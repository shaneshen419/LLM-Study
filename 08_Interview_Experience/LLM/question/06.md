## VLLM prefix cache的原理
vLLM的 Prefix Cache（前缀缓存）是一种用于大幅加速大语言模型推理、提升吞吐量并节省显存的核心优化技术。

它的核心思想非常直观：**如果多个不同的请求具有相同的前缀（例如相同的 System Prompt、Few-shot示例、或者 RAG 检索出的共享背景文档），vLLM会在显存中直接复用这部分前缀已经计算好的 KV Cache，从而跳过重复的 Prefill（预填充）计算**。

vLLM 实现 Prefix Cache 的底层原理主要依赖于 **Page Attention 架构**和 **Radix Tree（基数树）**。以下是其工作原理的详细拆解：

### 一、基础前提：Page Attention的块机制（Block）

在 vLLM 中，显存不再是连续分配的，而是像操作系统的虚拟内存一样，被划分成了固定大小的物理块。
- 默认情况下，一个 Block 存储 16 个token的 KV Cache。
- Prefix Cache 的匹配和复用，也是以 **Block 为最小力度**进行的，而不是单个 token。

### 二、核心数据结构：Radix Tree（基数树/字典树）

为了加速查找和匹配前缀，vLLM 在底层维护了一颗全局的 Radix Tree。
- **节点（Node）**：树的每一个节点代表一个或多个连续的 Token Block。
- **路径（Path）**：从根节点到某个节点的路径，代表了一段完整的 Prompt 前缀。
- **哈希映射**：每个 Block 内的 token  序列会被计算出一个 Hash 值，用于在树中快速进行分支路由和匹配。

### 三、Prefix Cache 的完整工作流程

当一个新的推理请求（Prompt）到达时，vLLM 会执行以下步骤：
- **第一步：分块与哈希计算**：将心情求的 Prompt Token 序列按照设定的 Block 大小（如16）进行切分，并计算每个逻辑块的 Hash 值。
- **第二步：树中匹配（prefix Matching）**拿着这些 Hash 值去全局的 Radix Tree 中自顶向下查找，寻找**最长匹配前缀**。
- **第三步：KV Cache 零拷贝复用**：对于匹配到的前缀块（A-B），vLLM 不会把它们送进大模型进行计算。相反，它直接在页表（Block Table）中，将当前请求的逻辑块映射到 Radix Tree 中已有的物理块上，并增加物理块的**引用计数（Reference Count）**。
- **第四步：增量计算**：对于未匹配的部分（C-D），vLLM 会正常分配新的物理块，并将其送入 GPU 进行前向传播计算得出 KV Cache。
- **第五步：更新树结构**：计算完成后，新生成的 KV Cache（C-D）会被作为新的子节点插入到 Radix Tree 中（挂在 A-B 节点下），供未来的请求使用。

### 四、显存管理与淘汰机制

GPU 的显存是有限的，Radix Tree 不可能无线生长。
- **引用计数**：只要一个 Block 正在被某个活跃的请求使用，它的引用计数就大于 0，绝对不会被释放。
- **LRU （最近最少使用）淘汰**：当显存不足，需要为新请求分配空间时，vLLM 会扫描 Radix Tree 中引用计数为 0 的节点，并按照 LRU 策略，优先剔除那些最长时间没有被访问过的前缀分支，回收其物理显存。

### 总结：Prefix Cache 解决了什么痛点？

1. **极大降低 TTFT（首字延迟）**： 对于长上下文的重复前缀，由于跳过了庞大的 Attention 矩阵乘法计算，首字生成时间可以从几秒骤降到几十毫秒。

2. **典型受益场景**：
    - **多轮对话**： 历史对话记录天然是共享前缀。
    - **Agent 智能体框架**： 多个 Agent 并发执行时，往往共享相同的超长 System Prompt。
    - **树搜索生成 (Tree of Thoughts / GRPO rollouts)**： 同一个 Prompt 采样多个不同的回答，Prompt 的 KV cache 可以 100% 复用。
